# Регистрация пользователей и админка

## Как это устроено

### 1. Хранение пользователей

- **Файл:** `data/users.json` (или каталог из `DATA_DIR`).
- Пароли хранятся в виде **хеша** (bcrypt), не в открытом виде.
- У каждого пользователя есть **роль:** `user` или `admin`.
- **Первый зарегистрированный пользователь** автоматически получает роль `admin`; остальные — `user`.

### 2. Сессии

- После входа сервер создаёт **сессию** (express-session).
- Идентификатор сессии хранится в **cookie** (`connect.sid`), с флагом `httpOnly` и `sameSite: "lax"`.
- Для запросов с фронта нужно отправлять cookie: во всех вызовах `fetch` к API используется `credentials: "include"`.

### 3. API авторизации

| Метод | Путь | Описание |
|-------|------|----------|
| POST | /api/auth/register | Регистрация: body `{ email, password }`. Пароль не короче 6 символов. |
| POST | /api/auth/login | Вход: body `{ email, password }`. В ответе — данные пользователя. |
| POST | /api/auth/logout | Выход: сессия уничтожается. |
| GET | /api/auth/me | Текущий пользователь (по сессии). 401, если не авторизован. |

### 4. Защита эндпоинтов

- **GET /api/labels** — только для **авторизованного администратора**. Обычный пользователь получит 403.
- **POST /api/label** (отправка разметки) — пока доступен всем; позже можно ограничить только авторизованными пользователями через middleware `requireAuth`.

### 5. Админка

- Страница **/admin.html** показывает все собранные записи разметки (таблица) и кнопку «Скачать CSV».
- Доступ к данным идёт через тот же **GET /api/labels** (с cookie), поэтому страница доступна только администратору.
- Если пользователь не залогинен или не админ — показывается сообщение «Войдите под учётной записью администратора».

## Как добавить функционал только для пользователей

1. **Требовать вход для действия**  
   На нужный роут повесьте middleware `requireAuth` (и при необходимости `requireAdmin`):

   ```ts
   router.post("/label", requireAuth, (req, res) => { ... });
   ```

2. **Использовать пользователя в логике**  
   В обработчике доступен `req.user`: `{ id, email, role }`. Например, сохранять в разметку `userId` или показывать в админке, кто отправил запись.

3. **На фронте**  
   Уже есть проверка: после загрузки страницы вызывается `GET /api/auth/me`; по ответу показываются кнопки «Войти/Регистрация» или email + «Выйти» + «Админка» (если админ). Для действий «только для своих» можно проверять `user` и при его отсутствии показывать форму входа или редирект.

---

## Переменные окружения

- **SESSION_SECRET** — секрет для подписи cookie сессии. В продакшене обязательно задать свой длинный случайный ключ.
- **DATA_DIR** — каталог для `users.json` и `user_labels.csv` (по умолчанию `./data`).

## Краткий сценарий проверки

1. Запустить бэкенд: `npm run dev`.
2. Открыть http://localhost:3000.
3. Нажать «Регистрация», ввести email и пароль (≥ 6 символов) — вы станете первым пользователем и получите роль admin.
4. Убедиться, что в шапке отображается email и ссылка «Админка».
5. Перейти в «Админка» — должна открыться таблица разметки (пока пустая или с уже имеющимися данными).
6. Скачать CSV через кнопку «Скачать CSV».
7. Выйти, зайти под другим пользователем (предварительно зарегистрированным) — ссылка «Админка» не отображается; при прямом переходе на /admin.html — сообщение о необходимости входа как администратор.
